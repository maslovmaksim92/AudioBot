# Политика предотвращения дубликатов файлов и регрессии кода

## Цель

Этот документ устанавливает правила и процедуры для предотвращения создания дублирующих файлов, конфликтующего кода и потери функциональности в проекте VasDom AudioBot.

## Проблемы, которые мы решаем

1. **Дублирование файлов** - создание нескольких версий одного и того же файла
2. **Конфликтующий код** - несколько реализаций одной функции
3. **Потеря функциональности** - случайное удаление или перезапись рабочего кода
4. **Несогласованность структуры** - разные подходы к организации кода

## Правила разработки

### ❌ ЗАПРЕЩЕНО

1. **Создавать альтернативные entry points**
   ```
   ❌ /app/backend/main.py
   ❌ /app/backend/app.py
   ❌ /app/backend/start.py
   ✅ /app/backend/server.py (единственный)
   ```

2. **Дублировать логику роутеров**
   ```
   ❌ /app/backend/routes/houses.py
   ❌ /app/backend/api/houses.py
   ✅ /app/backend/app/routers/houses.py (единственный)
   ```

3. **Создавать параллельные структуры компонентов**
   ```
   ❌ /app/frontend/src/pages/Dashboard.jsx
   ❌ /app/frontend/src/views/Dashboard.jsx
   ✅ /app/frontend/src/components/Dashboard/Dashboard.jsx
   ```

4. **Хранить секреты в коде**
   ```
   ❌ OPENAI_API_KEY = "sk-..."
   ❌ DATABASE_URL = "postgresql://..."
   ✅ os.environ.get('OPENAI_API_KEY')
   ```

5. **Коммитить .env файлы**
   ```
   ❌ git add backend/.env
   ✅ Используйте .env.example как шаблон
   ```

6. **Создавать временные файлы без очистки**
   ```
   ❌ test_temp.py, backup_old.jsx, server_v2.py
   ✅ Удаляйте временные файлы сразу после использования
   ```

### ✅ ОБЯЗАТЕЛЬНО

1. **Проверять существование файла перед созданием**
   ```bash
   # Перед созданием нового роутера
   find backend/app/routers -name "houses.py"
   ```

2. **Использовать модульную структуру**
   ```python
   # ✅ Правильно
   from app.routers import houses
   app.include_router(houses.router, prefix="/api")
   
   # ❌ Неправильно
   # Дублировать весь код в server.py
   ```

3. **Следовать принципу единственной ответственности**
   - Один файл = одна область ответственности
   - Один компонент = одна фича
   - Один роутер = один домен

4. **Документировать изменения**
   ```python
   # При изменении существующего файла:
   # HISTORY:
   # 2025-01-10: Добавлен фильтр по бригадам (Ivan Ivanov)
   # 2025-01-05: Создан файл (Original Author)
   ```

5. **Использовать git правильно**
   ```bash
   # Проверить изменения перед commit
   git diff
   git status
   
   # Убедиться, что не удалены важные файлы
   git diff --name-status
   ```

[ ] Добавил необходимые импорты из существующих модулей

## Чек-лист перед удалением файла

```
[ ] Убедился, что файл действительно не используется (grep по кодовой базе)
[ ] Проверил импорты в других файлах
[ ] Убедился, что есть альтернативная реализация (если файл не мусор)
[ ] Обновил документацию, если файл был там упомянут
[ ] Протестировал приложение после удаления
```

## Процедура рефакторинга

Если нужно изменить структуру проекта:

   - Документируйте текущую структуру

   - ARCHITECTURE.md
   - DEVELOPMENT_GUIDE.md
   - README.md

5. **Code review**
   - Pull request с детальным описанием
   - Review от другого разработчика
   - Проверка всех тестов

## Обнаружение дубликатов

### Автоматическая проверка

```bash
# Найти возможные дубликаты по именам файлов
find . -type f -name "*.py" | awk -F/ '{print $NF}' | sort | uniq -d

# Найти дубликаты по содержимому
fdupes -r backend/

# Проверить неиспользуемые импорты (Python)
vulture backend/

# Проверить мёртвый код (JavaScript)
npx depcheck frontend/
```

### Ручная проверка

При подозрении на дубликаты:

1. **Поиск похожих имён**
   ```bash
   find . -iname "*house*"
   find . -iname "*dashboard*"
   ```

2. **Проверка импортов**
   ```bash
   # Кто импортирует этот файл?
   grep -r "from app.routers import houses" backend/
   grep -r "import.*Houses" frontend/src/
   ```

3. **Сравнение содержимого**
   ```bash
   # Если есть подозрительные файлы
   diff -u file1.py file2.py
   ```

## Восстановление после ошибок

Если дубликаты уже созданы:

1. **Определите канонический файл**
   - Какой файл используется активно?
   - Какой файл имеет правильную структуру?
   - Какой файл полнее и актуальнее?

2. **Объедините функциональность**
   - Перенесите уникальные функции из дубликата
   - Удалите дублирующие функции
   - Обновите импорты

3. **Удалите дубликат**
   ```bash
   git rm path/to/duplicate/file.py
   git commit -m "fix: remove duplicate file, consolidated into canonical location"
   ```

4. **Обновите ссылки**
   ```bash
   # Найдите все места, где импортируется удалённый файл
   grep -r "from path.to.duplicate" .
   # Замените на правильный импорт
   ```

## Процедура для AI агентов

Если вы AI агент, работающий с этим проектом:

1. **Перед созданием файла**
   ```
   1. Проверьте ARCHITECTURE.md для канонического пути
   2. Используйте find/grep для поиска существующих файлов
   3. Если файл существует - модифицируйте его, НЕ создавайте новый
   ```

2. **При модификации**
   ```
   1. Прочитайте весь существующий файл
   2. Поймите текущую логику
   3. Добавьте/измените только необходимое
   4. Сохраните все существующие функции
   ```

3. **При удалении**
   ```
   1. Убедитесь, что файл действительно дубликат
   2. Проверьте, что все функции перенесены в канонический файл
   3. Обновите все импорты
   4. Протестируйте
   ```

4. **Документируйте изменения**
   ```
   1. Обновите комментарии в коде
   2. Обновите ARCHITECTURE.md если изменена структура
   3. Обновите API_DOCUMENTATION.md если изменены эндпоинты
   4. Добавьте entry в CHANGELOG.md
   ```
